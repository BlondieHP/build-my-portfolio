# Phase 4: 项目规约 (SPEC)

**目标**：将当前阶段的抽象知识映射为一个具体的、可完成的 demo 项目，产出 `projects/<slug>/SPEC.md`。

## 前置条件
- `.project-learning/ROADMAP.md` 已存在
- 已明确当前处于哪个里程碑
- 如果 ROADMAP.md 不存在，先跳回 [ROADMAP.md](ROADMAP.md)

## 流程

### Step 1: 确定项目定位
对照 ROADMAP.md 当前里程碑的学习目标，回答：
- 这个项目要练习哪 2-3 个核心概念？
- 做完这个项目，用户能说出"我学会了 X"吗？

### Step 2: 设计项目
遵循以下原则设计项目：

- **小而完整**：用户能在 2-4 小时内完成
- **刚好覆盖学习目标**：不引入超出当前里程碑的复杂度
- **有真实感**：即使是 demo，也要像一个真实场景而非教科书习题
- **可展示**：完成后推到 GitHub，README 让人看得懂

### Step 3: 写出项目结构
给出项目的目录结构。如果是程序员，给代码结构；如果是其他领域（设计、产品），给相应的产出物结构。

### Step 4: 定义验收标准
3-5 条明确的验收标准：
- 项目能独立跑起来（不必依赖 AI 上下文）
- 核心功能全部实现
- README 清晰可读

### Step 5: 用户确认
> 我为你设计了一个项目：[项目名称]
>
> **一句话描述**：用 X 技术做 Y，解决 Z 问题
> **练习的核心概念**：A, B, C
> **预计时间**：X 小时
> **验收标准**：[列出 3-5 条]
>
> 这个项目的难度和范围合适吗？

### Step 6: 写出 SPEC.md
按 [SPEC-FORMAT.md](../formats/SPEC-FORMAT.md) 格式写入 `.project-learning/projects/<project-slug>/SPEC.md`。

### Step 7: 脚手架（可选）
如果用户需要，运行 `scripts/scaffold-project.js` 生成项目骨架。
或者直接让用户自己创建——看用户偏好。

### Step 8: 交接
> 项目规约已经生成：[SPEC.md](.project-learning/projects/<slug>/SPEC.md)
>
> 下一步是**构建阶段**——在 AI 的辅助下，边学边做，把这个项目从零搭建起来。
>
> 准备好了就说"继续"，或者你也可以先自己看一下 SPEC 再开始。

## 项目设计原则
- **一个项目 = 一个可完成的挑战**：不是"做个淘宝"，而是"做个支持 Markdown 的便签应用"
- **技术选型从简**：用最主流、文档最完善的技术栈，不要让工具本身成为学习障碍
- **留白**：不要设计得太满，给用户留下自己探索和发挥的空间
- **失败友好**：如果项目太简单或太难，允许用户随时反馈调整
