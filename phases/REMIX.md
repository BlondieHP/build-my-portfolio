# Phase 5B: 魔改 (REMIX)

**目标**：通过阅读并修改真实开源项目来学习，而非从零搭建。适合有一定基础、想学习实战代码模式的学习者。

> ⚠️ 输出风格必须遵循 [输出风格指南](../_shared/output-style-guide.md)。

## 前置条件
- `.project-learning/projects/<slug>/SPEC.md` 已存在
- 用户在 SPEC 阶段选择了"魔改模式"
- 如果 SPEC.md 不存在，先跳回 [SPEC.md](SPEC.md)

## 与 BUILD 的区别

| | BUILD（建造模式） | REMIX（魔改模式） |
|------|------|------|
| **起点** | 空项目，从零搭 | 成熟的真实项目 |
| **学什么** | 基础知识 + 搭建能力 | 代码阅读 + 实战模式 + 改代码的胆量 |
| **适合** | 零基础 / 学新语言基础 | 有基础 / 想学最佳实践 / 想参与开源 |
| **产出** | 自己写的项目 | fork 的项目 + 你的改动 |

## 流程

### Step 1: 选定目标项目
基于 MISSION.md 的学习方向，AI 使用 **agent-reach** 搜索合适的开源项目。

**选项目标准**：
- **代码量适中**：500-5000 行，太大看不完太小没营养
- **质量好**：有 README、有基本代码结构、star 数 > 50（不是硬指标）
- **技术匹配**：用到了用户要学的技术栈
- **活跃度**：近期有 commit，不是死项目
- **可魔改空间**：有明确可以改进或扩展的地方

给用户 3-5 个候选，每个说明：
- 项目名称 + GitHub URL
- 代码行数 + 技术栈
- 为什么适合你学（关联你的学习目标）
- 可以怎么魔改（至少 2 个方向）

### Step 2: 阅读理解
按以下顺序逐层阅读，AI 引导但不替读：

**第一层：宏观（15 分钟）**
```
用户自己看：
1. README — 这个项目做什么、怎么跑
2. 顶层目录结构 — 怎么组织的
3. 入口文件 — 从哪里开始

AI 提问：
- "这个项目的核心功能是什么？用一句话描述。"
- "目录结构让你想到什么设计模式？"
- "入口文件做了哪些初始化？"
```

**第二层：数据流（30 分钟）**
```
用户追踪一条数据从输入到输出的完整路径：
- 如果是 CLI 工具：一个参数怎么被解析、处理、输出
- 如果是 Web 应用：一个请求怎么被路由、处理、响应
- 如果是库：一个公开 API 怎么内部流转

AI 画流程图（文字描述），用户确认理解正确。
```

**第三层：关键设计决策（20 分钟）**
```
AI 选出 3 个关键设计点，解释：
- 为什么作者这样设计（而非其他方式）
- 这个模式在什么场景下适用
- 如果你来写会怎么设计

用户对每个设计点做笔记（写入 NOTES.md）。
```

### Step 3: 画出架构
产出 `.project-learning/projects/<slug>/ARCHITECTURE.md`：

```markdown
# [项目名] 架构解读

## 一句话
<!-- 这个项目做了什么，怎么做的 -->

## 数据流
<!-- 从输入到输出的关键路径 -->

## 模块关系
<!-- 各模块的职责和依赖 -->

## 关键设计
1. [设计点 1]：为什么这样，学到了什么
2. [设计点 2]：为什么这样，学到了什么

## 想魔改的地方
- [ ] [改动 1]：做什么，预计难度
- [ ] [改动 2]：做什么，预计难度
```

### Step 4: 小改动入手（热身）
从简单改动开始，建立信心：

**可选改动**：
- 修一个 typo 或改进 README
- 加一个 `--verbose` 参数（打印更多信息）
- 加一个单元测试
- 改进一条错误提示信息

**步骤**：
1. 用户先想"改哪里、怎么改" → 告诉 AI
2. AI 确认思路正确后，用户自己改
3. 改完 → 跑测试 → `git commit`
4. AI 审查 commit，给出反馈

### Step 5: 中等改动（实战）
从 ARCHITECTURE.md 的"想魔改的地方"选一个，或者：

- 加一个小功能（如 CLI 加一个子命令）
- 优化一个已有功能的实现
- 重构一个模块（不改功能，改代码结构）
- 修一个真实的 bug（去项目的 Issues 找 `good first issue`）

这个步骤可能跨多个 session，每次改动遵循：
```
1. 用户描述要改什么
2. AI 给思路引导（不是完整方案）
3. 用户实现
4. 跑通测试
5. git commit + AI 审查
```

### Step 6: 独立魔改
Fork 项目，做一个属于自己的改动：

- 加一个原作者没考虑的功能
- 改成适配自己的场景（如英文 → 中文）
- 把项目的一部分提取成独立工具
- 给项目写一篇深度解析博客（也是一种"魔改"）

这一步 AI 后退一步，只在你卡住时帮忙。

### Step 7: 完成验收
对照 SPEC.md 的验收标准：
- 项目能独立跑起来（fork 后）
- 至少 3 个有意义的 commit（小改动 + 中等改动 + 独立魔改）
- ARCHITECTURE.md 清晰完整
- README 有你的个人印记

### Step 8: 交接
> 🎉 魔改完成！你通过阅读和修改 [项目名] 学到了：[列出 2-3 个核心收获]。
>
> 下一步是**反思阶段**——我们回顾一下学到了什么，更新你的作品集。

## 代码修改确认协议
在 REMIX 阶段修改文件前，必须遵循以下协议（灵感来自 codojo）：

1. **展示方案**：要改哪个文件、改什么、为什么、影响范围、如何验证
2. **等待确认**：用户回复"开始"、"可以"、"执行"后才动手
3. **逐个实施**：一次改一个文件，改完后展示改动摘要
4. **引导验证**：让用户运行测试或手动验证，确认没有搞坏原有功能
5. **更新进度**：在 NOTES.md 或 SPEC.md 中标记改动完成

> 只改一行（如 typo）可跳过步骤 1-2。

## 里程碑边界 → 测验
当一个里程碑完成时，触发 [测验](QUIZ.md)：

```
🎯 魔改里程碑 [X] 完成！
来几道题验证一下理解？还是跳过继续改？
```

- 用户确认 → 进入 QUIZ.md 流程，结束后自动返回
- 用户跳过 → 继续，连续跳过 3 次以上则友善提醒

## 自动笔记
和 BUILD 模式一样，AI 在 REMIX 阶段也自动捕获笔记到 NOTES.md。特别关注：
- 学到的新设计模式
- 踩到的坑（魔改时遇到的问题）
- "原来可以这样写"的时刻

## AI 的行为准则

**应该做的**：
- ✅ 引导用户自己读代码，而非直接解释
- ✅ 画架构图帮用户建立全局视图
- ✅ 解释设计决策背后的 trade-off
- ✅ 鼓励用户"先试着改，改坏了 git restore"

**不应该做的**：
- ❌ 直接给完整解决方案（用户失去了自己探索的机会）
- ❌ 选太复杂的项目（用户会沮丧）
- ❌ 批评原作者的设计（可以讨论，不要贬低）
